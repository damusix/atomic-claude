# Follow-ups folder + TTL

## Goal

Replace `.claude/project/followups.md` (single file, merge-conflict-prone) with `.claude/project/followups/<id>.md` (one entry per file) plus a regenerated `INDEX.md` for auto-load. Add TTL semantics so stale entries surface via `atomic doctor` and `/follow-up review`.

## Non-goals

- Auto-promoting stale entries to GitHub issues without user confirmation (only as an opt-in disposition during review).
- Deleting closed entries from git history (closure = file deletion; `git log` is the audit trail per existing convention).
- Migrating third-party follow-up systems (Linear, Jira) — out of scope; `Origin:` field stays free-form.
- Changing how scratchpad `FOLLOWUPS.md` works inside `/subagent-implementation` (that's working memory, distinct from project-level promotion target).

## Success criteria

- [ ] `.claude/project/followups/<id>.md` is the only place open project-level follow-ups live; `followups.md` deleted after migration.
- [ ] Every open entry file has YAML frontmatter with `id`, `title`, `created`, `origin`, `severity`, `review_by`, `status` fields. String values with colons / em-dashes / commas are quoted or use block-scalar form.
- [ ] `.claude/project/followups/INDEX.md` is regenerated by `atomic followups render`, listing one entry per line with severity + title + age in days.
- [ ] `.claude/project/followups/CLOSED.md` preserves a one-line audit-trail entry for every closed entry (title + closure marker), so closed entries can be deleted from the folder without losing rationale.
- [ ] Auto-load file `@`-refs `INDEX.md` instead of `followups.md` after migration. Search order matches `atomic-signals` precedent verbatim: `claude.local.md` / `CLAUDE.local.md` first (treated as a pair — whichever exists), then `claude.md` / `CLAUDE.md` (same pair semantics). Rewrite targets the exact `@.claude/project/followups.md` line — prose mentions of "followups.md" are not touched.
- [ ] `.githooks/pre-commit` regenerates `INDEX.md` when any `followups/*.md` file is staged and re-stages the index automatically. Hook degrades gracefully when `atomic` binary is absent (prints WARN, skips render step, commit still proceeds).
- [ ] `atomic followups add --id <id> --title <t> --severity <s> --origin <o> [--file <f>] [--body -]` deterministically creates a new entry file with valid frontmatter. `--body -` reads body from stdin (preferred for shell-pipe usage); without it the file is created with empty body for caller to append via Write tool. No LLM-authored frontmatter.
- [ ] `atomic followups list` prints all open entries; `--stale` filters to past-`review_by`; `--json` for machine output.
- [ ] `atomic followups close <id> [--reason <r>]` writes an entry to `CLOSED.md` (format includes the reason if supplied) then deletes `<id>.md` and regenerates INDEX.
- [ ] `atomic followups migrate` is idempotent: splits legacy `followups.md` into per-entry files + CLOSED.md, deletes the old file, no-ops if folder exists and old file absent.
- [ ] Migration refuses to delete `followups.md` when zero entries parsed (fails loud instead).
- [ ] `atomic doctor` followups check (replaced wholesale — see CP7) warns (not fails) on entries past `review_by` and on INDEX-out-of-sync. The current regex parser in `checks_followups.go` (which already silently misparses real headers because it anchors on literal `F-` while real entries are `<slug>-F-N`) is removed; new check reads frontmatter only.
- [ ] `/follow-up review` lists stale entries, per-item disposition: extend / close / promote / skip. Per-item confirm per axiom 3.
- [ ] `commands/subagent-implementation.md` defer disposition is rewritten to shell out to `atomic followups add` rather than writing to the legacy path.
- [ ] TTL default is 60 days, overridable via user memory (`feedback_followups_ttl` value format restricted to `Nd` — accept `60d`, `90d`, `30d`; reject `60 days`, `2w`, `1m`).
- [ ] All open entries in `followups.md` with slug-prefixed ids (e.g. `atomic-doctor-F-1`) survive migration with stable ids. **Nit-bucket bare `F-N` entries are NOT migrated by the tool** — migration refuses to invent a topic slug (that's an LLM judgment surface; spec is deterministic). Instead, migration prints the bare-id entries to stdout with their bodies and the user runs `atomic followups add` post-migration with hand-chosen ids. Reasoning preserved in the migration log; nothing lost.
- [ ] Migration supports `--dry-run` flag: prints every planned write (new files, CLOSED.md append, @-ref rewrite, legacy file deletion) and exits 0 without touching the filesystem.
- [ ] Migration requires `--accept-fallback-dates` when any entry's `Origin:` paragraph contains no parseable ISO date; otherwise migration refuses to proceed and lists the affected entries. Default behavior fails loud; opt-in flag accepts today as fallback `created` value.
- [ ] README and CLAUDE.md / CLAUDE.md updated to reference the new path + new CLI surface.

## Entry frontmatter schema

YAML scalars with colons, em-dashes, or commas must be quoted; long-form `origin` uses block scalar.

```yaml
---
id: atomic-doctor-F-1
title: "bundlemirror.Run double-reads files via path reconstruction"
created: 2026-05-17
origin: |
  docs/spec/atomic-doctor.md, iter 3 reviewer (CP-2). Deferred to project
  follow-ups at Phase 3 finalize 2026-05-17.
severity: risk
review_by: 2026-07-16
status: open
file: atomic/internal/bundlemirror/mirror.go:196-216
---

<entry body — free-form markdown, same content as today>
```

`atomic followups add` always emits quoted-title + block-scalar-origin. Hand-edited entries should follow suit, but the parser accepts any valid YAML.

### Field rules

| Field | Required | Validation | Notes |
|-------|----------|-----------|-------|
| `id` | yes | kebab-case, must match filename stem | Stable across renames; never reused on close |
| `title` | yes | non-empty string | Used in INDEX |
| `created` | yes | ISO date `YYYY-MM-DD` | Set at creation; never edited |
| `origin` | yes | free-form string | Where the finding came from |
| `severity` | yes | `risk` \| `nit` \| `question` | Mirrors current `🟡 risks` / `🔵 nits` / `❓ questions` headings |
| `review_by` | yes | ISO date | Default = `created + ttl_days`; `/follow-up review extend` updates this |
| `status` | yes | `open` \| `closed` | `closed` entries are deleted; this field exists for grep-ability during the migration window |
| `file` | no | path[:lines] | Optional file:line pointer |

## INDEX.md format

Regenerated by `atomic followups render`. Never hand-edited.

```markdown
# Project follow-ups index

Auto-generated by `atomic followups render`. Do not edit.

Open: 7  •  Stale: 2  •  Last rendered: 2026-05-21

## 🟡 risks (4)

- [atomic-doctor-F-1](atomic-doctor-F-1.md) — bundlemirror double-reads files (4d old)
- [atomic-doctor-F-2](atomic-doctor-F-2.md) — gitToplevel called 3× per run (4d, **stale**)
- ...

## 🔵 nits (3)

- [skill-routing-F-1](skill-routing-F-1.md) — atomic-tdd/atomic-debug boundary (4d)
- ...

## ❓ questions (0)

(none)
```

Stale = `today > review_by`. Header counts include stale.

## CLI surface

| Command | Behavior | Exit |
|---------|----------|------|
| `atomic followups list` | Print open entries grouped by severity | 0 |
| `atomic followups list --stale` | Open entries past `review_by` only | 0 |
| `atomic followups list --json` | Machine-readable | 0 |
| `atomic followups add --id <id> --title <t> --severity <s> --origin <o> [--file <f>] [--body -]` | Deterministic creation: validate args, compute `review_by`, write file with frontmatter (+ optional stdin body via `--body -`), regen INDEX, print path | 0; 1 on validation error |
| `atomic followups close <id> [--reason <r>]` | Append entry to `CLOSED.md` (with reason if supplied), delete `<id>.md`, regen INDEX | 0; 1 if id missing |
| `atomic followups render` | Regenerate INDEX.md from current folder contents | 0 |
| `atomic followups migrate [--dry-run] [--accept-fallback-dates]` | Split legacy `followups.md` → per-entry files + CLOSED.md; rewrite @-ref; render INDEX; delete old file. `--dry-run` prints plan only; `--accept-fallback-dates` required when any entry lacks a parseable date | 0; 2 on refusal; idempotent |
| `atomic followups path` | Print absolute path to the folder | 0 |

All commands resolve folder via the repo's `.claude/project/followups/` (cwd anchored to git toplevel, same as `signals` subcommand).

`add` is the deterministic-creation entrypoint: invoking artifacts (notably `/subagent-implementation` Phase 3 defer disposition) shell out to it rather than authoring frontmatter directly. Eliminates the hallucination surface — LLM only writes the body, never the structured fields.

## TTL semantics

- **Default TTL**: 60 days from `created`.
- **Memory override**: user can say "remember: follow-ups TTL is 90d" → saved as feedback memory `feedback_followups_ttl` with value `90d`. CLI reads this on every `render` and `list --stale`. **Value format restricted to `Nd`** (positive integer + literal `d`); other forms (`60 days`, `2w`, `1m`) are rejected with a parse error pointing back to the constraint.
- **Per-entry override**: user can hand-edit `review_by` in frontmatter; `render` respects it.
- **Stale ≠ expired**: stale entries warn but never auto-close. Closure requires explicit user disposition.
- **Renewal**: `/follow-up review` `extend` disposition rewrites `review_by` to `today + ttl_days`.
- **Age display**: INDEX always shows age in days (e.g. `4d`, `47d`, `120d`). No rollover to weeks/months — predictable golden tests, no unit-boundary edge cases.

## Defer disposition (new contract)

Replaces `commands/subagent-implementation.md` Phase 3 `defer` block (current ~5-step flow that writes to legacy `followups.md`). New flow:

```
1. Compute frontmatter args:
   --id       = "<topic-slug>-F-<N>"   # topic-slug from spec/design file path; N from existing count
   --title    = "<short finding description>"
   --severity = risk | nit | question  # from reviewer's tag
   --origin   = "docs/spec/<topic>.md, iter <N> reviewer (CP-<X>)"
   --file     = "<path:line>"  # optional pointer

2. Capture body via stdin pipe:
   <reviewer's finding text + recommendation> | atomic followups add \
       --id ... --title ... --severity ... --origin ... --file ... --body -

3. `atomic followups add` writes the file and regenerates INDEX.md.
   The @-ref already targets followups/INDEX.md (set during migration).

4. Stage + commit:
   git add .claude/project/followups/<id>.md .claude/project/followups/INDEX.md
   git commit -m "docs(followups): defer <id>"
```

No id-collision retry loop in the LLM layer — `add` rejects duplicate ids with exit 1 and the caller surfaces the rejection. No `@-ref` verification step — `migrate` already handled that one-shot. Two source-of-truth simplifications vs the legacy flow.

## Migration (one-shot)

`atomic followups migrate`:

1. Read `.claude/project/followups.md`.
2. Split on `### ` line-start markers. For each split, the first line is the header. Capture id + title by splitting the header on the first ` — ` or ` -- ` or ` - ` separator (whichever appears first). Everything until the next `### ` (or EOF) is the body.
3. Severity from enclosing `## 🟡 risks` / `## 🔵 nits` / `## ❓ questions` heading.
4. `origin`: full paragraph following the `Origin:` label up to the next blank line. Stored as block-scalar YAML to preserve commas/em-dashes/multi-line.
5. `created`: first ISO date `\d{4}-\d{2}-\d{2}` found in the origin paragraph (if multiple, first wins). If none parseable AND `--accept-fallback-dates` was supplied → today + WARN stderr listing the fallback entries. If none parseable AND the flag was NOT supplied → migration refuses (exit 2) and lists affected entries.
6. `review_by` = `created + ttl_days` (default 60).
7. `file`: optional pointer extracted from the first line of the body if it matches `` `<path>:<lines>` `` shape.
8. Closure detection: closure marker `*(closed YYYY-MM-DD ... )*` matched ONLY in the H3 header line (em-dash or hyphen accepted inside the parens; body text is ignored). Closed entries → CLOSED.md ledger line only; no live entry file.
9. **Bare-`F-N` entries (no slug prefix) are NOT migrated automatically.** Migration prints each bare-id entry's title + body to stdout under a "MANUAL MIGRATION REQUIRED" header, then exits 0 (or 2 in `--dry-run`). User picks slug+id and runs `atomic followups add` post-migration. No LLM judgment in the migration tool.
10. Write each migrated (slug-prefixed) open entry to `followups/<id>.md`. Write closed entries to `CLOSED.md` (append).
11. Rewrite `@-ref`: in the first auto-load file containing `@.claude/project/followups.md` (search order matches `atomic-signals` skill: `claude.local.md` / `CLAUDE.local.md` first, then `claude.md` / `CLAUDE.md`), replace that exact line with `@.claude/project/followups/INDEX.md`. Prose mentions of "followups.md" elsewhere in the file are not touched.
12. Run `render` to produce INDEX.md.
13. Delete `followups.md`.

**`--dry-run` mode** skips all filesystem writes (no entry files, no CLOSED.md append, no @-ref rewrite, no legacy deletion) and prints every planned operation to stdout in apply order. Exit 0 on clean plan; exit 2 if any refusal case would trigger.

**Refusal cases (fail loud, exit 2):**

- Both `followups/` and `followups.md` exist with entries in both → "manual resolution required".
- Zero entries parsed from `followups.md` → "no entries parsed; refusing to delete source".
- ID collision after re-namespacing → "duplicate ids: <list>".
- No `@-ref` to legacy path found in any auto-load file → "no @-ref found; rewrite would be silent; resolve manually".

**No-op case (exit 0):**

- `followups/` exists, `followups.md` absent → migration already ran.

## Pre-commit hook wiring

`.githooks/pre-commit` gains a new step (before bundle regen):

```bash
staged=$(git diff --cached --name-only)
if printf '%s\n' "$staged" | grep -qE '^\.claude/project/followups/[^/]+\.md$' \
   && ! printf '%s\n' "$staged" | grep -q '^\.claude/project/followups/INDEX\.md$'; then
  if command -v atomic >/dev/null 2>&1; then
    atomic followups render
    git add .claude/project/followups/INDEX.md
  else
    echo "warning: atomic binary not on PATH; skipping follow-ups INDEX regen" >&2
  fi
fi
```

Notes:

- Style adjusted to existing hook conventions (`set -euo pipefail` already in effect). `printf | grep` pattern avoids subshell exit triggering pipefail abort.
- Negation guards against re-firing when INDEX is the only thing staged.
- `command -v atomic` fallback: fresh clones or pre-install states get a WARN, not a failed commit.
- INDEX.md is not bundled — `bundlemirror` sources from `agents/`, `commands/`, `skills/`, `output-styles/`, `rules/`, `CLAUDE.md` only; `.claude/project/` is not in scope. Ordering "before bundle regen" is fine.

## Doctor integration

**Replace** `atomic/internal/doctor/checks_followups.go` wholesale. The existing implementation parses the legacy markdown file with a regex anchored on `^### F-...`, which is already broken on the real file (real entries start with `<slug>-F-N`, not bare `F-`). Symptom today: check is silently no-op — entries unmatched, malformed states never surface. Replacing rather than amending closes that pre-existing bug as part of this work.

New check walks `.claude/project/followups/`, reads each file's frontmatter via the new `atomic/internal/followups` package, computes severity from frontmatter (not markdown bucket inference):

| Condition | Severity | Repair |
|-----------|----------|--------|
| `followups/` missing AND legacy `followups.md` present | WARN | `atomic followups migrate` |
| `followups/INDEX.md` missing or out of sync (byte-compare against re-rendered INDEX in memory) | WARN | `atomic followups render` |
| One or more entries past `review_by` | WARN | none auto; surfaces "/follow-up review" suggestion |
| One or more entries with invalid/missing frontmatter | WARN | none auto; surfaces failing file path |
| Folder absent AND no legacy file | SKIP | n/a |
| All entries fresh, INDEX in sync | PASS | n/a |

Doctor never auto-closes or auto-migrates without `--fix` confirmation.

## `/follow-up review` subverb

New disposition mode for `commands/follow-up.md`:

```
/follow-up review
```

Behavior:

1. Run `atomic followups list --stale --json`.
2. If empty → "no stale follow-ups; <N> open, all within TTL" + exit.
3. Print numbered list (per axiom 4: plain-text indexed selection).
4. Prompt: per-item disposition `extend | close | promote | skip`.
5. Apply each disposition:
   - `extend` → rewrite `review_by` to `today + ttl_days`, save file.
   - `close` → `atomic followups close <id>`.
   - `promote` → `gh issue create --title "<title>"` with body linking to origin, then close.
   - `skip` → no-op.
6. After all dispositions, regenerate INDEX.

Per-item confirm per axiom 3 (destructive); `close` and `promote` require explicit Yes.

## Auto-load `@-ref`

After migration, the chosen file (per the search order in success criteria) gets:

```markdown
## Project follow-ups (auto-loaded)

@.claude/project/followups/INDEX.md
```

Migration's rewrite step is exact-line anchored: it matches `@.claude/project/followups.md` literally and replaces only that line. Prose mentions of "followups.md" in surrounding text are left alone. If no auto-load file contains the legacy `@-ref` line, migration refuses to proceed (per Refusal cases in § Migration) — no silent rewrite, no implicit choice between candidate files.

## Risks

| Risk | Likelihood | Mitigation |
|------|-----------|-----------|
| Migration loses content for entries with non-conforming headers in legacy file | medium | Parser fails loud on unknown shape; refuses to delete source when zero entries parsed; user fixes legacy file by hand and re-runs |
| Closed-entry rationale lost on migration | medium | CLOSED.md ledger preserves title + closure marker as a one-liner; full body is in git log for the legacy `followups.md` deletion commit |
| INDEX.md drifts when users hand-edit entry files outside the hook | medium | `atomic doctor` followups check flags INDEX-out-of-sync; repair = `render`; pre-commit hook regenerates on staging |
| `git mv` to rename `id` breaks links from origins | low | `id` rename is rare; spec body states "treat id as stable"; entries that need renaming get closed + recreated |
| Memory-override TTL silently outlives a config change | low | Doctor's followups check prints effective TTL + source (memory vs default) in `--verbose` |
| `gh issue create` failures during `promote` leave entry in limbo | low | Promotion is two-step: create issue first; only on success close entry. On failure, entry stays open + WARN printed |
| Pre-commit hook double-fires render when INDEX is also being intentionally hand-edited | low | Hook checks if INDEX is already staged before re-rendering |
| Concurrent worktree edits to different entries collide on INDEX | low | INDEX regen is deterministic from folder contents; resolve by re-running `render` on the merge result; per-entry files don't conflict with each other |
| `atomic-followups-add` validation rejects an id that the user wants for a defer entry | low | Error message includes the validation rule (kebab-case, unique); caller (`/subagent-implementation`) prompts user for a new id and retries |
| Bare-`F-N` entry user-side re-add is forgotten after migration | medium | Migration prints bare-id entries with their bodies to stdout under "MANUAL MIGRATION REQUIRED" header so they don't silently vanish; user has the content in hand for `atomic followups add` |
| Closure `--reason` not preserved if user runs `close` without it | low | CLOSED.md format accommodates reason or marker; empty reason produces a marker-only line; encourage `--reason` in `/follow-up review` UI |

## Change log

### 2026-05-22 — No-backwards-compatibility simplifications applied during implementation

**What changed:** Implementation iteration 2 dropped the safety scaffolding the spec originally required for one-shot migration. Affected surfaces:

- `atomic followups migrate`: no `--dry-run` flag and no `--accept-fallback-dates` flag. `migrate` is one-shot, fail-loud. Refusal cases trimmed to two: zero entries parsed → exit 2; id collision after synthesis → exit 2. Dropped the "both `followups/` and `followups.md` exist with entries" refusal — we own the one legacy file and the migration owns its disposition.
- Bare-`F-N` legacy entries (nits bucket pre-migration) are migrated inline with synthesized topic-slug ids derived from the bucket name + slug-ified first words of the title. The "MANUAL MIGRATION REQUIRED" stdout block + user-side `atomic followups add` re-add dance is removed.
- The `@-ref`-absent → refuse case is dropped. Migration rewrites the legacy `@-ref` line if found and silently no-ops if not. The auto-load search order remains `claude.local.md` → `CLAUDE.local.md` → `claude.md` → `CLAUDE.md`.

**Why:** User directive mid-implementation ("don't worry about backwards compatibility"). The legacy `.claude/project/followups.md` exists in exactly one repo (this one), and the migration runs exactly once. Defensive flags for staged rollouts and partial-recovery scenarios paid no benefit and added test surface.

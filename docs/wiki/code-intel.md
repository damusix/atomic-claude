# code-intel

## What it does

Tree-sitter/wazero-based code-intelligence engine embedded in the [`atomic`](../../atomic) CLI. Parses source files into a SQLite-backed symbol graph, resolves cross-file references, and exposes the graph via the `atomic code` CLI verbs and an MCP server. Index lives at `<projectRoot>/.claude/.atomic-index/atomic.db`.

## CLI code

- [`atomic/internal/codeintel/cli/code.go`](../../atomic/internal/codeintel/cli/code.go) — `RunCode` dispatcher for `atomic code <verb>`: index, sync, status, search, callers, callees, impact, node, files, affected, explore, mcp. Internal `__serve` verb spawns the daemon.
- [`atomic/internal/codeintel/engine/engine.go`](../../atomic/internal/codeintel/engine/engine.go) — `Engine` facade (CP20): wraps db, pool, orchestrator, resolution pipeline, graph manager, searcher, and context builder. Both CLI and MCP compile against this.
- [`atomic/internal/codeintel/tsbinding/`](../../atomic/internal/codeintel/tsbinding) — wazero binding that loads `lib/ts.wasm` (embedded via `//go:embed`) and exposes tree-sitter parse/query APIs. Owns a separate `go.mod` (`module github.com/malivvan/tree-sitter`, go 1.23.4).
- [`atomic/internal/codeintel/extraction/extractor.go`](../../atomic/internal/codeintel/extraction/extractor.go) — `TreeSitterExtractor`: drives one file through the grammar, produces `ExtractionResult{Nodes, Edges, UnresolvedReferences}`. Appendix-E contract.
- [`atomic/internal/codeintel/extraction/languages/`](../../atomic/internal/codeintel/extraction/languages) — per-language `LanguageExtractor` configs: Go, TypeScript, TSX, JavaScript, Python, Ruby, Rust, Java, C, C++, C#, Swift, Kotlin, Dart, Lua, Luau, Objective-C, PHP, Pascal, Scala, plus a `registry.go`.
- [`atomic/internal/codeintel/extraction/standalone/sql.go`](../../atomic/internal/codeintel/extraction/standalone/sql.go) — SQL language extractor (standalone; not tree-sitter-backed). Parses DDL/DML to extract table, view, procedure, trigger, and function nodes; Snowflake dialect: CREATE STAGE (A5), CREATE STREAM (A4), CREATE TASK with AFTER predecessor edges (A3), COPY INTO body-edge lineage including @stage sigil tokens, CLONE references, class/security preamble modifiers. dbt model extraction (B section): Jinja pre-pass harvests `ref()` / `source()` into a new `model` node and reference edges; placeholder-substituted residual SQL feeds the standard extractor. **v2 additions**: `CREATE FILE FORMAT` → `NodeKindFileFormat` (O1); VARIANT/OBJECT/ARRAY column typing (O2); `LATERAL FLATTEN(INPUT => tbl)` arg reference (O4); dbt macro definitions `{% macro … %}…{% endmacro %}` → `NodeKindMacro` nodes (E2); macro-body ref attribution — refs inside a macro span own the macro node (E3); project-path role detection (`/macros/` → macro-only, `/tests/`|`/analyses/` → no model node) (O9); versioned-ref distinct targets `__dbt_ref_<model>_v<N>` (O7); `config(alias=...)` stored in model node Metadata (O8); standalone top-level COPY INTO with lazy `NodeKindScript` owner (F4); Jinja `{# … #}` comment-leak fix (strip before harvest).
- [`atomic/internal/codeintel/extraction/standalone/embedded_sql.go`](../../atomic/internal/codeintel/extraction/standalone/embedded_sql.go) — CP1: admission gate (`IsSQLLiteral`) and entry point (`ExtractEmbeddedSQL`) for embedded SQL extraction. Called by host-language harvesters; stamping `Provenance:"embedded"` on all produced edges.
- [`atomic/internal/codeintel/extraction/standalone/exts.go`](../../atomic/internal/codeintel/extraction/standalone/exts.go) — `SQLExtensions` canonical slice (`.sql`, `.ddl`, `.pgsql`, `.mysql`, `.sql.jinja`). Single source of truth for extension routing in orchestrator and resolution pipeline.
- [`atomic/internal/codeintel/extraction/python_literals.go`](../../atomic/internal/codeintel/extraction/python_literals.go) — CP3 (extraction half): tree-sitter-based Python string literal harvester. Returns spans with `IsDocstring` flag; f-string interpolations substituted to `"?"`.
- [`atomic/internal/codeintel/extraction/typescript_literals.go`](../../atomic/internal/codeintel/extraction/typescript_literals.go) — CP4 (extraction half): tree-sitter-based TypeScript/TSX string literal harvester. Template-literal `${...}` interpolations substituted to `"?"`.
- [`atomic/internal/codeintel/extraction/embedded_literals.go`](../../atomic/internal/codeintel/extraction/embedded_literals.go) — generic config-driven string-literal harvester for the 16 new host languages (CP2 generic path). `HarvestEmbeddedLiterals` accepts an `EmbeddedLiteralConfig` (StringKinds/ContentKinds/InterpKinds) and applies Shape 1 (content-child grammars) or Shape 2 (inline-content grammars). Returns `[]EmbeddedSpan` (mirrors `standalone.StringLiteralSpan` field-for-field; separate type to avoid import cycle).
- [`atomic/internal/codeintel/extraction/helpers.go`](../../atomic/internal/codeintel/extraction/helpers.go) — `GenerateNodeID` (exported stable-id formula: `kind + ":" + hex(sha256("filePath:kind:name:line"))[:32]`; file-nodes use `"file:" + filePath`). Extracted from `extractor.go` so sibling packages (e.g. `extraction/standalone`) can generate IDs without importing extraction internals.

- [`atomic/internal/codeintel/db/`](../../atomic/internal/codeintel/db) — SQLite layer: `db.go` enforces single-connection + 7-pragma appendix-O order; `schema.sql` embedded via `go:embed`; CRUD, migrations, stats, search, resolution helpers.
- [`atomic/internal/codeintel/indexer/orchestrator.go`](../../atomic/internal/codeintel/indexer/orchestrator.go) — `Orchestrator`: `IndexAll`, `IndexPaths`, `Sync` — walks project files, dispatches to pool, persists results. `IndexAll` and `Sync` call `pruneDeleted` after indexing to remove rows for files that no longer exist on disk. `compoundExt` helper extracts `.sql.jinja` as a compound suffix (stdlib `filepath.Ext` returns `.jinja` for compound extensions).

After host tree-sitter extraction, harvests string literals, gates via `IsSQLLiteral`, resolves owner host node, calls `ExtractEmbeddedSQL`, merges into `ExtractionResult`. Controlled by `embeddedSQLHostExts` registry.

## Coupling

- [`atomic/cmd/atomic/main.go`](../../atomic/cmd/atomic/main.go) dispatches `"code"` to `codecli.RunCode` — any change to the CLI surface requires updating `main.go:codeAction` and the `atomic code` help text.
- [`atomic/internal/codeintel/tsbinding/`](../../atomic/internal/codeintel/tsbinding) is a separate Go module (`go.mod`) — changes to wazero or tree-sitter version require a `go mod tidy` there independently.
- [`atomic/internal/codeintel/db/schema.sql`](../../atomic/internal/codeintel/db/schema.sql) is the schema source of truth — schema changes require a migration in `migrations.go` and version bump in `db.go`.
- Framework route extraction (`resolution/frameworks/`) runs after `IndexAll` and before `ResolveReferences` — pipeline ordering in `cli/code.go:runIndex` and `engine.go:ExtractFrameworkNodes` must stay in sync with spec appendix ordering.
- MCP tool names use `atomic_code_*` prefix — any rename breaks agents or IDE integrations that use the MCP protocol.
- [`scripts/code-eval/`](../../scripts/code-eval) eval harness (fetch-corpus + run-eval + embedded-sql-eval) exercises the full extraction pipeline; changes to extraction output format invalidate corpus results. `embedded-sql-eval.sh` added for gate-accuracy evaluation.
- [`atomic/cmd/embedded-sql-admission/`](../../atomic/cmd/embedded-sql-admission) — standalone eval binary; scans a directory tree and reports all string literals that pass the `IsSQLLiteral` admission gate. Not part of the main [`atomic`](../../atomic) binary.
- Index lives at `<projectRoot>/.claude/.atomic-index/` (gitignored via `EnsureGitignore`). Doctor check 11 (`checks_code_index.go`) reports index health — absence is informational PASS; stale is WARN; fresh is PASS. The check imports `engine.IndexPath`; if the path convention changes, both must change together.
- `engine.IndexPath` is the canonical function for deriving the DB path from a project root — consumers outside the engine package must call this function, not construct the path inline.
- **Agent consumption**: all keystone agents (investigator, builder, surgeon, reviewer, haiku, signals-inferrer) plus strategist carry the `agent-code-intel` partial. The partial contract: index present → lead with `atomic code explore "<query>"` for orientation (returns bundled context digest); then targeted verbs (`search`/`callers`/`callees`/`impact`) for specific symbol questions; bounded queries only (one explore question or one symbol at a time); graceful degradation to sg/grep on any failure (binary absent, DB missing, query error) — never surface errors about the index being unavailable.

## Conventions worth knowing

- **Branding hard rule**: the reference TypeScript engine's product name must never appear in code, comments, identifiers, strings, tool names, or output. Brand slug = [`atomic`](../../atomic); MCP tools = `atomic_code_*`; data dir = [`.claude/.atomic-index/`](../../.atomic-index).
- **Engine lifecycle**: `New` → (`Init` or `Open`) → use → `Close`. `Init` is idempotent; `Open` errors if index absent. All facade methods call `requireDB()` first.
- **Single-connection SQLite**: `SetMaxOpenConns(1)` + `SetMaxIdleConns(1)` enforced in `db/db.go`. Appendix-O pragma order is mandatory — `busy_timeout` first.
- **Extraction is best-effort**: errors are recorded in `ExtractionResult.Errors`, never abort. Unresolved calls emit `UnresolvedReference` (not edges); edges are created only after resolution.
- **Framework extraction ordering**: `ExtractFrameworkNodes` must run after `IndexAll`/`Sync` and before `ResolveReferences` so route→handler refs exist when the resolution pipeline runs.
- **Deleted-file pruning**: `pruneDeleted` (orchestrator) runs after every `IndexAll` and `Sync`. Builds an O(1) set of on-disk relative paths, queries `GetAllFiles`, then for each absent file runs one `WithTx` that calls `Tx.DeleteNodesByFile` (FK-cascade removes edges) + `Tx.DeleteUnresolvedRefsByFile` + `Tx.DeleteFile`. `IndexPaths` does NOT prune — it receives an explicit subset and would wrongly delete every out-of-subset file.
- **`engine.NewWithDBPath(projectRoot, dbPath)`**: overrides the default index-path convention to an explicit db path. Used by the MCP daemon for cwd-independent operation (`atomic --repo <path> code mcp`); a realm member daemon receives the realm db path directly so it never needs to resolve from cwd.
- **Watch stubbed in v1**: `Watch`/`StopWatch` return `ErrWatchNotImplemented` per appendix M.
- **SQL extractor is standalone**: `extraction/standalone/sql.go` does not use tree-sitter; it has its own parse logic. Reused verbatim by `embedded_sql.go` for DDL literals. Taxonomy: 38 `NodeKind` values (31 → 35 after Snowflake/dbt v1; 35 → 38 after v2; v2 T-SQL gaps add no new kinds). The seven SQL-dialect kinds: `stage`, `stream`, `task`, `model`, `file_format`, `macro`, `script`. `TestNodeKindCount` in `types/types_test.go` asserts the exact count (currently 38) — update when adding node kinds.
- **T-SQL lineage gaps v2 (proc-scoped temp tables, OUTPUT INTO, PIVOT, column-level)**: `sql.go` gains four capabilities. (1) Temp tables/table variables: `CREATE TABLE #tmp` / `SELECT … INTO #tmp` / `DECLARE @t TABLE(…)` emit `writes`+`references` edges to a synthetic routine-scoped node whose `Name` encodes the routine and whose `Metadata` carries `{"temp":"local"|"local_tvar"|"global"}` and the written token; scalar `DECLARE @id INT` emits nothing; global `##g` is file-level deduped. (2) `INSERT/UPDATE/DELETE/MERGE … OUTPUT <list> INTO <target>` emits a `writes` edge to `<target>`; match is bounded by output-list char class so a statement boundary prevents a false cross-statement match; `OUTPUT` without `INTO` emits no edge. (3) `PIVOT`/`UNPIVOT`: source `references` edge preserved via FROM clause; no edge to PIVOT/aggregate/IN-list values (confirmed no-op; test committed). (4) Column-level 4a: alias→table map built from FROM/JOIN clause; qualified `alias.col` emits a `references` edge to the column node (`acct.id` → `acct.id`); unqualified refs skipped. Two SQL-scoped resolver tweaks in `resolution/name_matcher.go`: (a) single-dot SQL ref routes to `byQualifiedName` (not bare-name lookup); (b) `byQualifiedName` prefers an exact full-QName candidate over partial matches. Pipeline pre-filter fix in `resolution/pipeline.go`: `hasAnyPossibleMatch` adds an SQL-scoped simple-name fallback so qualified column refs (`dbo.Account.account_id`) are not dropped before reaching `byQualifiedName`.
- **Resolver tweak SQL scope gate**: all three resolution changes (`name_matcher.go` tweak-a/b, `pipeline.go` pre-filter fallback) are gated on `ref.Language == LanguageSQL` — non-SQL `receiver.method` resolution is unchanged; existing cross-language resolution tests stay green.
- **Snowflake / dbt conventions**: `modPat` in `sql.go` absorbs `OR REPLACE` / `OR ALTER` / `IF NOT EXISTS` — do not re-add these in new regex patterns. dbt `ref('pkg','model')` has the package as the **first** positional; a bare 2nd string literal is the model name only when 2 positionals are supplied. Version is always a keyword arg (`v=`/`version=`), never positional. Project-path role (`dbtFileRole`): path segment `/macros/` → `"macro"` (no model node, harvest macro defs); `/tests/`|`/analyses/`|`/seeds/`|`/snapshots/` → `"other"` (no model node); all other paths → `"model"`. Macro-body attribution: refs inside `{% macro %}…{% endmacro %}` byte spans own the macro node (E3) — checked via `inMacroSpan` linear search (dbt does not support nested macros). Jinja `{# … #}` block comments stripped (`dbtJinjaCommentRE`) before harvest so comment-embedded refs are not extracted. `config(alias=...)` stored as JSON in model node `Metadata` field. Standalone top-level `COPY INTO` outside a dbt context creates a `NodeKindScript` owner named `<file-basename>` (lazy — created on first COPY found in file).
- **`.sql.jinja` compound-extension routing**: `filepath.Ext` returns `.jinja` for `stg.sql.jinja`, which would route to unknown. `compoundExt` helper in `orchestrator.go` checks `HasSuffix(".sql.jinja")` before falling through to `filepath.Ext`. `exts.go:SQLExtensions` includes `.sql.jinja` so the admission guard and extension map agree.
- **Embedded SQL pipeline**: `embeddedSQLPostPass` (orchestrator post-pass) → per-language literal harvester → `IsSQLLiteral` gate → `ExtractEmbeddedSQL` → merge into `ExtractionResult`. Hosts: Go (hand-written scanner), Python (tree-sitter, docstrings excluded), TypeScript/TSX (tree-sitter, template-literal interpolations substituted); **16 additional languages** via the generic `HarvestEmbeddedLiterals` harvester (C, C++, C#, Java, JavaScript, Kotlin, Lua, Luau, Objective-C, Pascal, PHP, Ruby, Rust, Scala, Swift, Dart) driven by per-language `EmbeddedLiteralConfig` in `indexer/embedded_literals_config.go`. Two extraction shapes: Shape 1 (content-child grammars — `string_content`-style children) and Shape 2 (inline-content grammars — take node's own text, splice `"?"` over interpolation byte ranges, strip delimiter-alphabet prefix/suffix). All embedded edges carry `Provenance:"embedded"` (queryable via `GetEdgesByProvenance("embedded")`). Standalone `.sql`/`.ddl`/`.pgsql`/`.mysql` routing is unchanged. Host-ext set derived from `extToLanguage` + config table (single source, not a second hand-maintained ext list).
- **Admission gate `IsSQLLiteral`**: canonical implementation in `standalone/embedded_sql.go`; recognizes SQL structure only (no stopword lists); multi-fragment / concatenated queries are accepted false negatives.
